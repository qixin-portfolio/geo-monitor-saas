# AI 协作协议（ChatGPT + AI Executor + GitHub）

## 核心原则

仓库是长期记忆，聊天只是临时界面。

所有任务、交付、审查、风险说明，都必须沉淀到 GitHub Issue、PR、
PR 评论和 `AI_TASKS/handoff.md`。

用户不再负责在 ChatGPT 和 AI Executor 之间复制长报告。

## AI Executor 执行代理

本项目的执行代理不限定为 Codex。

可用执行代理包括：

* **Codex**：适合 GitHub 仓库任务、代码修改、PR 创建、远程协作。
* **WorkBuddy**：适合本地项目查找、本地文件整理、本地构建、静态产物打包、上传前检查。
* 其他 AI 编程代理（Claude Code、Cursor Agent 等）：只要遵守本协议，也可以作为执行代理。

统一称呼：`AI Executor`。

下文出现“AI Executor”时，对 Codex / WorkBuddy / 其他代理均适用；
出现“Codex”或“WorkBuddy”时，仅指该代理的专属职责。

## WorkBuddy 角色定位

WorkBuddy 是本地执行代理，适合处理：

* 查找本地已有项目
* 检查本地目录结构
* 判断项目类型
* 修改静态页面或配置文件
* 执行本地构建
* 准备上传产物
* 检查是否误改其他项目
* 输出本地交付报告

WorkBuddy 不应负责：

* 自动合并 PR
* 自动发布生产环境
* 自动修改真实数据库
* 自动修改支付、认证、安全配置
* 自动处理真实密钥
* 绕过用户确认执行高风险操作

## 角色分工

### 用户

* 只提出目标
* 只提供 Issue / PR 编号或本地交付路径
* 只做最终合并、暂停或回滚决策
* 不负责搬运 AI Executor 长报告
* 不在聊天中粘贴密钥、token、数据库连接串

### ChatGPT

* 负责拆解任务
* 负责设计 Issue 内容
* 负责审查 PR 或本地交付报告
* 负责判断风险
* 负责给下一步策略
* 以 GitHub Issue / PR / handoff 为主要上下文

### AI Executor（Codex / WorkBuddy / 其他）

通用职责：

* 负责读取 Issue 或用户本地指令
* 负责在分支或本地工作区执行
* 负责修改代码或文件
* 负责运行检查
* 负责创建 PR 或输出本地交付报告
* 负责更新 PR 描述（如适用）
* 负责在 PR 评论或交付报告中写交付说明
* 负责更新 `AI_TASKS/handoff.md`

Codex 专属：

* 基于 GitHub Issue 新建分支
* 创建 PR、更新 PR 描述与评论

WorkBuddy 专属：

* 在本地工作区执行
* 输出本地交付路径、变更清单和影响范围说明
* 不直接合并 PR、不发布生产环境

## 标准流程

### 远程流程（Codex 等远程执行代理）

1. 用户提出目标
2. ChatGPT 输出 GitHub Issue 内容
3. 用户或 Codex 创建 Issue
4. Codex 基于 Issue 新建分支
5. Codex 执行任务
6. Codex 创建 PR
7. Codex 更新 PR 描述、PR 评论和 `AI_TASKS/handoff.md`
8. 用户只告诉 ChatGPT：看 PR #x
9. ChatGPT 审查 PR
10. 用户决定 merge / 修改 / 暂停

### 本地流程（WorkBuddy 等本地执行代理）

1. 用户提出目标（可能是远程 Issue 对应的本地实现，也可能是纯本地任务）
2. WorkBuddy 确认当前目录与目标项目，不得误改其他项目
3. WorkBuddy 输出允许修改范围
4. WorkBuddy 执行任务（如能使用 Git，必须走分支和 commit）
5. WorkBuddy 运行本地构建 / 检查
6. WorkBuddy 输出修改文件清单、变更摘要、是否影响其他项目
7. 如能推送，WorkBuddy 推送分支并创建 PR，再走远程流程的审查
8. 如不能推送，WorkBuddy 输出可上传目录和变更摘要，交由用户或 ChatGPT 审查
9. 用户决定合并 / 上传 / 修改 / 暂停

## 禁止流程

* 禁止 AI Executor 只在聊天中汇报，不写 PR 或本地交付报告
* 禁止用户复制 AI Executor 长报告给 ChatGPT
* 禁止 ChatGPT 长期输出大段让用户来回搬运的指令
* 禁止没有 Issue 或明确本地指令就直接做复杂功能
* 禁止把部署、数据库、支付、认证混在一个任务里
* 禁止提交真实密钥
* 禁止把晟景装饰内容混进 GEO Monitor SaaS
* 禁止 WorkBuddy 自动合并 PR、自动发布生产环境、自动修改真实数据库
* 禁止 WorkBuddy 在发现多个疑似项目时继续执行而不让用户选择

## Maker / Checker 分离

Codex、WorkBuddy 或其他 AI Executor 都属于 Maker。

Maker 负责：

* 执行任务
* 修改文件
* 运行检查
* 创建 PR 或输出本地交付报告
* 更新 `AI_TASKS/handoff.md`

Checker 负责：

* 审查结果
* 判断是否越界
* 判断是否可合并
* 判断是否进入 Human Gate
* 判断是否需要回滚或补修

默认 Checker 是 ChatGPT + 用户。

禁止 Maker 自己宣布任务最终完成。
Maker 只能说明“已完成执行并等待审查”。

## 本地任务协议

当任务由 WorkBuddy 或其他本地执行代理执行时，必须额外遵守：

1. 开始前先确认当前目录。
2. 修改前先确认目标项目，不得误改其他项目。
3. 如果发现多个疑似项目，必须停止并让用户选择。
4. 修改前输出允许修改范围。
5. 修改后输出修改文件清单。
6. 必须说明是否影响其他项目。
7. 如能使用 Git，必须走分支和 commit。
8. 如不能使用 Git，必须输出可上传目录和变更摘要。
9. 不得把真实密钥写入文件。
10. 不得直接删除重要文件。

## 最小沟通格式

用户以后只需要说：

* 创建任务：`我要做 xxx`
* Codex 执行：`让 Codex 继续 Issue #x`
* WorkBuddy 执行：`让 WorkBuddy 找本地项目并准备上传`
* ChatGPT 审查：`看 PR #x`
* 本地交付审查：`WorkBuddy 已完成，路径是 xxx`
* 暂停任务：`暂停 Issue #x`
* 合并后：`PR #x 已合并`
* 新开对话恢复：`继续这个项目，先读 AI_TASKS/PROTOCOL.md 和 AI_TASKS/LOOP_PROTOCOL.md`

## 每个任务的完成定义

任务只有满足以下条件才算完成：

1. GitHub Issue 已记录任务目标，或用户已给出明确本地指令
2. AI Executor 已创建 PR 或输出本地交付报告
3. PR 描述或本地交付报告完整
4. 自测结果写清楚
5. `AI_TASKS/handoff.md` 已更新
6. Checker（默认 ChatGPT + 用户）审查通过
7. 用户合并、上传或明确暂停

## 新对话恢复方式

任何新 ChatGPT 或 AI Executor 对话开始时，只需要输入：

“继续 geo-monitor-saas 的协作流程，请先根据仓库里的
AI_TASKS/PROTOCOL.md、AGENTS.md、PR 和 handoff 继续。”

ChatGPT 应优先基于仓库文件和 GitHub PR / Issue 状态继续，而不是依赖旧聊天内容。

## Loop 协议引用

* 当任务具有重复性、可验收、有价值时，必须参考
  `AI_TASKS/LOOP_PROTOCOL.md` 设计任务循环。
* 部署、数据库、认证、支付、批量内容生成、GEO 监控、CI 修复等任务，
  默认需要判断是否进入 Loop 模式。
* 任何 Loop 都必须有停止条件、验收标准、状态文件和 Human Gate。
* ChatGPT 负责判断任务是否应该进入 Loop。
* AI Executor 不得擅自把任务变成无限循环。
* WorkBuddy 在本地执行 Loop 任务时，同样必须遵守停止条件、Human Gate 和状态文件规则。

## AI 输出质量原则

本项目默认要求 AI 在复杂任务中使用更高质量的分析框架，而不是只给表面答案。

### 1. 第一性原理

复杂问题应优先从第一性原理出发，拆解到最基本的事实、约束、
目标和不可变条件，避免只依赖惯例、类比或已有方案。

适用场景：

* 产品定位
* 架构设计
* 商业模式判断
* 重大技术方案选择
* 成本、风险、收益分析

输出要求：

* 先说明核心问题是什么
* 再说明不可变约束是什么
* 最后从约束重新推导可执行方案

### 2. 系统思维

涉及多个模块、角色、流程或长期影响的任务，应使用系统思维分析。

需要关注：

* 系统边界
* 关键角色
* 输入与输出
* 反馈循环
* 杠杆点
* 可能的副作用
* 一阶、二阶和三阶影响

适用场景：

* SaaS 功能规划
* 用户增长系统
* 支付、认证、数据库、部署流程
* AI 协作流程
* GEO 内容工程

### 3. 关键推理摘要，而不是完整隐藏思考链

AI 不需要展示完整隐藏思考链，但必须展示足够的关键推理摘要，
方便用户和后续 AI 审查判断。

输出应包含：

* 结论
* 判断依据
* 关键取舍
* 风险点
* 反例或失败条件
* 下一步动作

禁止要求：

* 不要求输出完整隐藏 Chain of Thought
* 不把内部推理过程当成交付物
* 不用长篇推理替代可执行结果

### 4. 苏格拉底式质疑与反向思维

对重要方案，AI 应主动挑战假设。

需要检查：

* 当前假设是什么
* 如果相反情况成立会怎样
* 什么因素会导致方案失败
* 哪些风险被低估
* 有没有更简单的替代方案

适用场景：

* PR 审查
* 上线前检查
* 商业方案判断
* 架构重构
* 成本投入决策

### 5. 默会知识与现场经验

AI 在给建议时，应结合领域专家的隐性经验，而不是只输出书面知识。

要求：

* 识别真实现场里容易出问题的环节
* 说明有经验的人会如何判断
* 提醒那些文档里不明显但实践中高频踩坑的问题
* 把显性规则和现场经验结合起来

适用场景：

* 装修行业产品
* 本地商家获客
* SaaS 销售转化
* 小程序上线
* Vercel / Clerk / Stripe / 数据库部署

### 6. 多维度与第二序效应

复杂任务不能只看眼前结果，还要分析后续影响。

至少从以下维度中选择相关维度分析：

* 技术
* 成本
* 用户体验
* 商业价值
* 品牌信任
* 合规风险
* 维护成本
* 长期扩展性

需要说明：

* 直接后果
* 后果的后果
* 哪些选择短期舒服但长期麻烦
* 哪些选择短期麻烦但长期稳定

### 7. 费曼式表达

对用户输出时，应尽量用简单、清楚、可执行的话解释复杂问题。

要求：

* 不堆术语
* 不写空泛正确的话
* 不故作复杂
* 能用一句话讲明白就先讲一句话
* 再给结构化步骤

### 8. 默认结构化输出

复杂任务默认按以下结构输出：

1. 核心结论
2. 判断依据
3. 可执行步骤
4. 风险与反例
5. 更好替代方案
6. 下一步动作

如果是 PR 审查，默认输出：

1. 是否可以合并
2. 是否越界
3. 是否有安全风险
4. 是否有部署风险
5. 是否需要补 commit
6. 给 AI Executor 的最小修复要求

如果是 WorkBuddy 本地交付审查，默认输出：

1. 是否只改了允许范围
2. 是否误改其他项目
3. 是否有真实密钥泄露风险
4. 本地构建是否通过
5. 是否需要补 commit 或重新打包
6. 是否可以进入上传 / 合并阶段

### 9. 轻重分级

不是每个小任务都需要完整展开。

简单任务：

* 直接给结论和步骤

中等任务：

* 给关键依据、风险和操作步骤

复杂任务：

* 使用第一性原理、系统思维、反向思维和多维度分析

避免为了“看起来高级”而制造冗长输出。
