# Loop Engineering Protocol：循环工程执行协议

> 目标：把重复性、可验收、高价值任务设计成可重复、可验证、可停止、可追踪、可追责的工程循环。

## 1. 什么时候使用 Loop

任务同时满足以下三个条件时，必须优先设计 Loop：

1. 重复性：未来会反复出现。
2. 可验收：可以明确写出完成标准。
3. 有价值：产出价值高于执行时间、token、风险成本。

如果任务不可验收、只做一次、或高度依赖主观判断，不应直接做成自动循环，应采用人工审查或单次辅助模式。

## 2. Loop 六个组件

每个 Loop 必须定义：

### A. Trigger / 心跳

说明任务如何启动：

- 手动触发。
- 定时触发。
- 事件触发。
- 目标驱动触发。

### B. Goal / 目标

说明本轮循环最终要达成什么。

### C. Acceptance Criteria / 验收标准

写清楚什么结果算完成，例如：

- `pnpm typecheck` 通过。
- `pnpm build` 通过。
- 所有改动仅限指定目录。
- 不修改生产环境配置。
- 不直接推送 `main`。

### D. Execution Boundary / 执行边界

明确 AI 可以做什么、不能做什么。高风险动作必须交给 Human Gate。

### E. Reviewer / 独立检查者

Maker 不允许只靠自我判断宣布完成。必须至少通过以下一种方式验证：

- 自动测试。
- linter / typecheck / build。
- reviewer agent。
- 人工审查。

### F. State File / 状态文件

跨轮次任务必须维护状态文件，例如：

- `AI_TASKS/progress.md`
- `AI_TASKS/loop-state.md`
- `AI_TASKS/release-log.md`

## 3. 做 / 检分离

复杂任务拆成两个角色：

- Maker：负责执行、修改、生成、修复。
- Checker：负责验证、审查、挑错、判断是否达标。

Checker 重点检查：

1. 是否满足原始目标。
2. 是否通过测试或构建。
3. 是否引入额外副作用。
4. 是否修改不该修改的文件。
5. 是否存在安全、数据、权限、生产环境风险。

## 4. 停止条件

任何 Loop 都必须有硬刹车，至少设置一种，推荐三种都设置：

- 次数上限：例如最多 5 轮。
- 时间上限：例如最多 30 分钟。
- 成本上限：例如超过预算、token、API 调用限制即停止。

遇到以下情况必须停止并记录原因：

- 连续多轮没有实质进展。
- 需要生产密钥、生产库、付费外部 API。
- 需要删除或批量改写真实数据。
- 需要修改支付、认证、权限、安全策略。
- 工作区出现大量意外变更。
- 构建或测试失败且无法在边界内修复。

## 5. Human Gate

以下任务不得由 AI 自动完成最终操作：

1. 生产部署。
2. 生产数据库迁移。
3. 删除、重置、批量改写数据。
4. 修改支付、认证、权限、安全策略。
5. 自动合并 PR。
6. 自动发布正式版本。
7. 批量发送邮件、短信、消息或营销内容。

AI 可以准备方案、生成 diff、创建草稿 PR、输出检查报告，但必须等待用户确认。

## 6. 状态文件格式

Loop 状态文件至少包含：

```md
## 已完成

- 日期：完成了什么，关联 commit / PR / 文件。

## 进行中

- 当前任务卡在哪里，下一步是什么。

## 需要人类判断

- 风险项。
- 不确定项。
- 无法自动验证的事项。

## 本轮运行记录

- 执行了哪些命令。
- 修改了哪些文件。
- 通过了哪些检查。
- 失败了哪些检查。
```

## 7. 默认执行要求

1. 先理解目标，再执行。
2. 先列验收标准，再改文件。
3. 优先小步修改，不做大而全重构。
4. 不直接动 `main`。
5. 不擅自删除文件。
6. 不擅自改环境变量、密钥、部署配置。
7. 所有高风险操作必须停止并请人确认。
8. 修改后必须记录改了什么、为什么改、如何验证、是否还有风险、下一步建议。
