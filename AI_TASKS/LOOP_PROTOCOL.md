# Loop Engineering Protocol：循环工程执行协议

> 本协议用于所有需要 AI 长链路执行、反复修改、自动检查或持续维护的任务。
> 核心原则：不要只写一次性 prompt，而要设计一个可重复、可验证、可停止、
> 可追踪的任务循环。

## 1. 什么时候必须使用 Loop

当任务同时满足以下三个条件时，必须优先设计 Loop，而不是只进行单轮对话：

1. **重复性**：该任务未来会反复出现，值得沉淀为流程。
2. **可验收**：可以明确写出“完成长什么样”。
3. **有价值**：产出价值高于执行所消耗的时间、token 和风险成本。

如果任务不可验收、只做一次、或者结果高度依赖主观判断，则不得直接做成自动循环，
应采用人工审查或单次辅助模式。

## 2. 每个 Loop 必须包含六个组件

任何 AI 自动执行任务，至少要定义以下六项：

1. **Trigger / 心跳**
   说明任务如何启动：手动触发、定时触发、事件触发，还是目标驱动触发。

2. **Goal / 目标**
   说明这次任务最终要达成什么。

3. **Acceptance Criteria / 验收标准**
   必须写清楚“什么结果算完成”。

   示例：

   * `npm run build` 通过
   * `npm run typecheck` 通过
   * 所有改动仅限指定目录
   * 不修改生产环境配置
   * 不直接推送 main 分支

4. **Execution Boundary / 执行边界**
   明确 AI 可以做什么、不能做什么。
   高风险动作必须交给人类确认。

5. **Reviewer / 独立检查者**
   执行任务的 AI 不允许自己给自己判定成功。
   必须通过测试、linter、reviewer agent 或人工检查之一完成验证。

6. **State File / 状态文件**
   对于跨轮次、跨天、跨分支任务，必须维护状态文件。

   推荐文件：

   * `AI_TASKS/progress.md`
   * `AI_TASKS/release-log.md`
   * `AI_TASKS/loop-state.md`

## 3. 做/检分离原则

所有复杂任务必须拆成两个角色：

* **Maker**：负责执行、修改、生成、修复。
* **Checker**：负责验证、审查、挑错、判断是否达标。

Codex、WorkBuddy 或其他 AI Executor 都属于 Maker。
默认 Checker 是 ChatGPT + 用户。

禁止 Maker 自己宣布任务完成。
Maker 只能说明“已完成执行并等待审查”。

Checker 需要重点检查：

1. 是否满足原始目标。
2. 是否通过测试或构建。
3. 是否引入额外副作用。
4. 是否修改了不该修改的文件。
5. 是否存在安全、数据、权限或生产环境风险。

WorkBuddy 作为本地 Maker 时，Checker 还需额外检查：

* 是否误改其他项目。
* 是否在发现多个疑似项目时停止并让用户选择。
* 是否输出修改文件清单和影响范围说明。
* 是否把真实密钥写入了文件。

## 4. 每个 Loop 必须有硬刹车

任何循环任务不得无限执行。必须至少设置以下一种上限，推荐三种都设置：

1. **次数上限**
   示例：最多尝试 5 轮。

2. **时间上限**
   示例：最多运行 30 分钟。

3. **成本上限**
   示例：超过预算、token 或 API 调用限制即停止。

如果连续多轮没有实质进展，必须停止，并输出失败原因与下一步建议。

## 5. 高风险任务必须进入 Human Gate

以下任务不得由 AI 自动完成最终操作，必须交给人类确认：

1. 直接修改生产环境。
2. 删除数据、迁移数据库、重置配置。
3. 修改支付、权限、登录、安全策略。
4. 自动合并 PR。
5. 自动发布正式版本。
6. 批量发送邮件、消息或营销内容。
7. 任何影响真实用户、真实资金、真实业务的数据操作。

AI 可以准备方案、生成 diff、创建草稿、输出检查报告，但不能绕过人类闸门。

## 6. 状态文件规则

每次 Loop 开始前，必须先读取状态文件。
每次 Loop 结束后，必须更新状态文件。

状态文件至少包含：

```md
## 已完成

- 日期：完成了什么，关联 commit / PR / 文件

## 进行中

- 当前任务卡在哪里，下一步是什么

## 需要人类判断

- 风险项
- 不确定项
- 无法自动验证的事项

## 本轮运行记录

- 执行了哪些命令
- 修改了哪些文件
- 通过了哪些检查
- 失败了哪些检查
```

如果 AI 多次犯同一个错误，不要只修改 prompt，而要把规则沉淀进协议文件或 skill 文件。

## 7. 推荐执行模式

### 模式 A：单次任务模式

适合小任务。

```md
目标：
验收标准：
允许修改范围：
禁止事项：
完成后输出：
```

### 模式 B：Goal Loop 模式

适合测试修复、代码重构、内容批处理。

```md
目标：
循环条件：
停止条件：
最多尝试次数：
每轮必须验证：
失败后如何记录：
```

### 模式 C：Daily Maintenance Loop 模式

适合每日维护、CI 检查、issue 分诊、GEO 内容监控。

```md
触发时间：
读取状态：
检查来源：
处理优先级：
自动处理范围：
需要人类确认的情况：
最终输出：
```

## 8. AI 执行任务时的默认要求

除非特别说明，所有 AI 任务默认遵守：

1. 先理解目标，再执行。
2. 先列验收标准，再改文件。
3. 优先小步修改，不做大而全重构。
4. 不直接动 main / master。
5. 不擅自删除文件。
6. 不擅自改环境变量、密钥、部署配置。
7. 所有高风险操作必须停止并请人确认。
8. 修改后必须说明：

   * 改了什么。
   * 为什么改。
   * 如何验证。
   * 是否还有风险。
   * 下一步建议。

### 本地任务额外要求（WorkBuddy 等本地执行代理）

1. 开始前先确认当前目录。
2. 修改前先确认目标项目，不得误改其他项目。
3. 如果发现多个疑似项目，必须停止并让用户选择。
4. 修改前输出允许修改范围。
5. 修改后输出修改文件清单。
6. 必须说明是否影响其他项目。
7. 如能使用 Git，必须走分支和 commit。
8. 如不能使用 Git，必须输出可上传目录和变更摘要。
9. 不得把真实密钥写入文件。
10. 不得直接删除重要文件。

## 9. 最终原则

AI 的价值不是替人逃避理解，而是把已经理解、重复、可验证的工作加速。

人类负责：

1. 定义意图。
2. 设定边界。
3. 制定验收。
4. 承担结果。

AI 负责：

1. 执行重复步骤。
2. 检查明显问题。
3. 起草方案。
4. 记录状态。
5. 把风险事项交还给人。

任何 AI Loop 都必须遵守：

> 可重复、可验证、可停止、可追踪、可追责。
